# 전략 파라미터 패널 & 코드 백그라운드화 개선안

> **구현 상태 (일부)**: API `POST /api/strategies/params/extract|apply`, 웹 우측 패널 기본 탭「파라미터」, 릴레이 생성 프롬프트에 `STRATEGY_PARAMS` 규약, 예시 `rsi_long_short_strategy.py` 반영.

> 목적: 비개발자 사용자가 **파이썬 코드를 읽거나 직접 편집하지 않고**, 생성된 전략의 **조절 가능한 값(숫자·선택값 등)** 만 UI에서 안전하게 바꿀 수 있게 한다.  
> 실행 코드는 항상 존재하되 **UI 기본 경로에서는 노출하지 않는다.**

---

## 1. 원칙

| 원칙 | 설명 |
|------|------|
| 단일 진실 원천 | 런타임(백테스트/라이브)은 여전히 **전략 파이썬 모듈**을 로드한다. |
| UI 분리 | 사용자에게 보이는 기본 화면은 **요약 + 스펙 + 파라미터 폼**. 코드 편집기는 **고급/옵션**. |
| 안정적 패널 | 전략 로직이 달라도 패널 구동 방식은 동일하게 하기 위해 **선언적 파라미터 스키마**가 필수다. |
| 저장 전 검증 | 파라미터 반영 후에도 기존과 같이 **문법 검증·(가능하면) 검증용 백테스트**를 통과해야 저장된다. |

---

## 2. 데이터 모델 (핵심)

### 2.1 전략 파일 측 규약

생성 전략(또는 템플릿)에 아래 중 **하나의 고정 규약**을 둔다.

- **권장**: 모듈 최상위에 `STRATEGY_PARAMS: dict[str, Any]` 와 동일 키를 참조하는 로직만 사용.  
- **보강**: UI·검증용으로 `STRATEGY_PARAM_SCHEMA` (JSON Schema 호환 dict 또는 Pydantic 모델 직렬화)를 같이 두거나, 생성 시 **스키마 JSON을 별도 필드**(DB/메타)에 저장.

스키마 예시(개념):

```json
{
  "rsi_period": { "type": "integer", "min": 2, "max": 100, "default": 14, "label": "RSI 기간" },
  "rsi_oversold": { "type": "number", "min": 1, "max": 50, "default": 30, "label": "과매도 기준" }
}
```

- `type`: `integer` | `number` | `boolean` | `string` (필요 시 `enum`)  
- UI는 스키마만 보고 입력 위젯을 결정한다.

### 2.2 서버/DB 메타 (선택)

- `StrategyMeta` 또는 blob 외에 **`param_schema` + `param_values` JSON** 캐시를 두면, 코드 파싱 없이 패널을 그릴 수 있다.  
- 단, **최종 일치**는 “코드 내 `STRATEGY_PARAMS` ↔ 저장된 값” 동기화 정책으로 정한다 (아래 4절).

## 3. 사용자 경험 (웹 Strategies 워크스페이스)

### 3.1 우측 패널 구조

- **기본 탭**: `요약` + `파라미터` (스키마 기반 폼: 슬라이더, 숫자 입력, 토글, 선택).  
- **고급 탭 (접기/토글)**: 기존 **Workspace Code** (`textarea`, 문법 검사, 초기 대비 diff).  
- **설정 옵션 (후순위)**: “코드 워크스페이스 표시” 사용자 설정으로 고급 탭 자체를 숨김.

### 3.2 상호작용

1. 전략 생성 완료 또는 목록에서 불러오기 → 서버/클라이언트가 **스키마 + 현재 값**을 로드.  
2. 사용자가 폼에서 값 변경 → **미리보기 영역**에 “진입: RSI(14) < 30 …”처럼 사람이 읽는 문장 갱신(선택, LLM 또는 템플릿).  
3. 적용 시:  
   - **A안 (클라이언트)**: 현재 `workspaceCode` 문자열에 대해 안전한 치환(아래 4.2).  
   - **B안 (서버)**: `POST`로 `{ code, params }` 전달 → 서버가 AST/템플릿으로 패치 후 새 코드 반환.  
4. 기존 `validate-syntax` 및 `save` 전 검증 플로우 유지.

### 3.3 스키마 없는 구전략

- 패널에 **“이 전략은 조절 가능한 파라미터 정의가 없습니다”** 안내.  
- **폴백**: 채팅으로 수정 유도 또는 **“스키마 추출”** 버튼( LLM 1회로 `STRATEGY_PARAMS`+스키마 제안 → 사용자 확인 후 삽입).

---

## 4. 파라미터 ↔ 코드 동기화 전략

### 4.1 생성 단계 (LLM / 릴레이)

- 시스템 프롬프트·템플릿에 **필수 규약** 명시:  
  - 모든 튜닝 가능한 상수는 `STRATEGY_PARAMS["키"]` 로만 참조.  
  - 스키마(또는 주석 블록)를 파일 상단 또는 메타에 포함.  
- 검증 스크립트(`strategy-verify` 스킬 등)에 **“PARAMS 블록 존재 및 스키마 키와 키 일치”** 검사 추가를 검토.

### 4.2 수정 단계 (권장 순서)

1. **서버 측 패치 (권장)**: 입력 검증된 `param_values`만으로 `STRATEGY_PARAMS` dict 리터럴을 치환하거나 AST로 안전 갱신 → 코드 재출력.  
2. 클라이언트만으로 할 경우: 정규식 치환은 깨지기 쉬우므로 **최소한 서버에서 한 번 더 검증** 권장.

### 4.3 채팅 기반 수정과의 관계

- **숫자·임계값만**: 파라미터 패널이 1순위.  
- **로직 변경**: 채팅 재생성. 재생성 후에도 동일 규약으로 스키마를 유지하거나 재생성되면 스키마 병합 정책 필요.

## 5. API 개편 (제안)

| 엔드포인트 | 역할 |
|------------|------|
| `GET /api/strategies/.../params` (또는 내용 응답에 포함) | 현재 코드에서 추출한 `schema` + `values` |
| `POST /api/strategies/apply-params` | `{ code, param_values }` → `{ code }` (검증 optional) |
| 기존 `POST /api/strategies/validate-syntax`, `save` | 변경 후 그대로 사용 |

(실제 URL은 기존 `getStrategyContent` / `saveStrategy` 설계에 맞춰 통합 가능.)

---

## 6. 구현 단계 (로드맵)

| 단계 | 내용 |
|------|------|
| **P0** | 릴레이/템플릿에 `STRATEGY_PARAMS` + 스키마 생성 규약 문서화 및 샘플 전략 1~2개 적용 |
| **P1** | 백엔드: 코드에서 params 추출(안전 로드/AST) + `apply-params` 초안 |
| **P2** | 웹: 우측 패널 탭화 — 기본은 파라미터 폼, 코드는 고급 |
| **P3** | 메타 DB에 schema/values 캐시, 목록 로딩 최적화 |
| **P4** | 구전략 스키마 추출( LLM ), 사용자 토글 “코드 숨기기” |

---

## 7. 리스크 및 완화

- **스키마와 코드 불일치**: 저장 시 `STRATEGY_PARAMS` 재파싱 후 스키마 키와 교차 검증.  
- **악의적 코드**: params 패치는 **신뢰된 생성 파이프**를 전제로 하고, 임의 업로드는 기존 검증·샌드박스 유지.  
- **UX**: 패널 적용 실패 시 명확한 에러 메시지 + 채팅 폴백 안내.

---

## 8. 현재 코드베이스와의 연결

- **프론트**: `web/src/app/strategies/page.tsx` — 우측 `Workspace Code` aside를 탭 구조로 확장하는 진입점.  
- **저장/검증**: `src/api/main.py`의 `save_strategy`, `validate-syntax` — 파라미터 적용 후 동일 경로 통과.  
- **생성 품질**: `relay` 프롬프트·템플릿(`scripts/strategies` / indicator strategy 스킬)에 규약 반영.

---

## 9. 결론

**코드는 백그라운드**로 두고, 사용자에게는 **선언된 스키마 기반 파라미터 패널**을 기본으로 제공한다.  
안정성은 **무한한 자유 코드 파싱**이 아니라 **`STRATEGY_PARAMS` + 스키마 규약**으로 확보하고, 저장·실행 파이프라인은 기존과 동일하게 **검증된 파이썬**을 유지한다.
